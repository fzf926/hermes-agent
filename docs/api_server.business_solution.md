# Hermes 智能问答与 SQL 收藏能力方案说明

## 一、方案定位

Hermes API Server 是面向业务系统的智能数据分析服务。它把“自然语言提问、自动执行 SQL、沉淀历史记录、收藏高价值结果、基于收藏继续分析”串成一个完整闭环。

对业务用户来说，它不是一个普通接口集合，而是一个“懂数据库的智能助理”。用户不需要反复找人写 SQL，也不需要记住每一次查询的上下文。系统会自动理解问题、执行查询、总结结果，并把有价值的分析过程沉淀下来。

本方案覆盖以下核心能力：

- 智能问答：`POST /v1/responses`
- 模型发现：`GET /v1/models`
- SQL 收藏：`POST /api/chat/favorites`
- 收藏列表：`GET /api/chat/users/{userId}/favorites`
- 会话详情：`GET /api/chat/sessions/{sessionId}/turns`
- 历史会话：`GET /api/chat/users/{userId}/sessions`

## 二、建设目标

本能力建设的目标，是让数据查询从“一次性问答”升级为“可沉淀、可追溯、可复用的数据分析工作流”。

系统希望解决以下问题：

- 用户不知道怎么写 SQL，也能用自然语言完成数据查询。
- 每次问答都自动保存，用户可以回看历史分析过程。
- 重要 SQL 结果可以收藏，后续无需重新查找或重新提问。
- 收藏结果可以作为新的分析起点，继续追问和扩展分析。
- 问答、SQL、工具调用都有记录，方便追溯、排查和运营分析。

## 三、整体能力闭环

完整用户流程如下：

1. 用户在前端输入问题，例如“查询今天订单数”。
2. 前端调用 `POST /v1/responses`。
3. Hermes 理解问题，并判断是否需要执行 SQL。
4. 系统调用后端工具完成 SQL 查询。
5. Hermes 将查询结果整理成自然语言答案返回给用户。
6. 系统自动保存本次问答、SQL 执行记录和工具调用记录。
7. 用户认为结果有价值时，点击“收藏”。
8. 前端调用 `POST /api/chat/favorites` 创建 SQL 收藏。
9. 用户之后进入“我的收藏”，查看历史收藏结果。
10. 用户可以基于某条收藏继续追问，形成连续分析链路。

最终形成：

```text
用户提问
  ↓
Hermes 理解与推理
  ↓
SQL 执行与结果整理
  ↓
答案返回
  ↓
历史沉淀
  ↓
高价值结果收藏
  ↓
基于收藏继续分析
```

## 四、核心能力说明

### 1. 智能问答能力

接口：`POST /v1/responses`

这是整个系统的核心入口。前端把用户输入的问题发送给 Hermes，Hermes 负责完成理解、推理、SQL 执行、结果总结和响应返回。

用户看到的是自然语言问答体验。系统背后会自动完成：

- 理解用户问题
- 判断是否需要查询数据库
- 调用 SQL 执行能力
- 整理查询结果
- 生成清晰答案
- 保存本次问答过程

该接口支持两种返回方式：

- 普通响应：适合一次性返回完整结果。
- 流式响应：适合像 ChatGPT 一样边生成边展示，用户等待体验更好。

业务价值：

- 降低数据查询门槛。
- 提升数据获取效率。
- 将用户问题、答案和 SQL 结果自动沉淀。
- 支撑后续历史回看、收藏和继续追问。

### 2. 模型发现能力

接口：`GET /v1/models`

该接口用于告诉前端当前可用的 Hermes 模型名称。业务用户通常不会直接感知它，但它是前端接入智能问答能力的基础接口。

典型用途：

- 前端初始化时获取可用模型。
- 兼容 OpenAI 风格 SDK 或调用方式。
- 支持后续多模型展示或模型切换。

业务价值：

- 让前端不用写死模型名称。
- 为未来模型升级、模型切换预留扩展空间。
- 降低前后端集成成本。

### 3. SQL 收藏能力

接口：`POST /api/chat/favorites`

当用户对某次 SQL 查询结果满意时，可以一键收藏。收藏的不是一段普通文本，而是一组经过系统整理的结构化资产。

收藏内容包括：

- 用户当时问了什么
- Hermes 如何回答
- 实际执行了哪些 SQL
- SQL 是否执行成功
- SQL 查询耗时
- 查询结果是否满足用户问题
- 后续是否已经基于该收藏继续追问

为了保证收藏质量，系统会做准入校验：

- 必须存在真实 SQL 执行记录。
- SQL 必须执行成功。
- SQL 查询耗时不能超过 5 秒。
- 查询结果必须属于当前用户。
- 同一用户重复收藏同一结果时，不会重复创建。

业务价值：

- 用户可以沉淀高价值查询结果。
- 高频分析场景可以复用，不必反复提问。
- 收藏结果可以成为后续分析的起点。
- 帮助产品形成“个人数据知识库”能力。

### 4. 用户收藏列表能力

接口：`GET /api/chat/users/{userId}/favorites`

该接口用于展示某个用户收藏过的 SQL 问答结果。前端可以基于它构建“我的收藏”页面。

列表中会展示：

- 收藏时间
- 问题摘要
- 回答摘要
- SQL 数量
- 原始会话信息
- 后续追问会话信息

业务价值：

- 用户可以快速找回过去有价值的数据分析。
- 收藏内容不只是结果截图，而是可继续使用的分析资产。
- 为后续“常用查询”“分析模板”“智能推荐”提供数据基础。

### 5. 会话详情能力

接口：`GET /api/chat/sessions/{sessionId}/turns`

该接口用于查看某一次会话中的完整问答过程。

一次会话可能包含多轮问答，例如：

1. 用户询问“查询上周订单量”。
2. Hermes 返回订单总量和趋势。
3. 用户继续追问“按城市拆分”。
4. Hermes 继续执行 SQL 并返回城市维度结果。

会话详情会返回：

- 每轮用户问题
- 每轮 Hermes 回答
- 每轮执行过的 SQL
- 每轮工具调用记录
- 查询是否成功
- 回答是否满足问题
- 每轮对应的响应 ID

业务价值：

- 用户可以完整回看一次分析过程。
- 复杂分析不再丢上下文。
- 出现争议或问题时，可以追溯当时系统如何回答、执行了什么 SQL。
- 为客服、运营、数据团队排查问题提供依据。

### 6. 用户历史会话能力

接口：`GET /api/chat/users/{userId}/sessions`

该接口用于展示某个用户的历史会话列表。前端可以基于它构建“历史记录”页面。

列表中会展示：

- 会话 ID
- 会话类型
- 最近一次问题
- 问答轮数
- 创建时间
- 更新时间

系统当前支持三类会话：

- 普通历史问答
- 基于收藏继续追问
- 直接 SQL 查询

默认展示普通历史问答和直接 SQL 查询。收藏续聊可以按需单独筛选。

业务价值：

- 用户可以按时间找回历史分析。
- 产品可以围绕历史记录构建“继续对话”体验。
- 后台可以基于历史会话分析用户常见数据需求。

## 五、前端产品形态建议

建议前端围绕以下页面组织能力。

### 1. 智能问答页

用户输入自然语言问题，系统实时返回 Hermes 的分析结果。

适合展示：

- 用户输入框
- 流式回答
- SQL 执行状态
- 查询结果摘要
- 收藏按钮

### 2. 历史会话页

展示用户过去所有会话，帮助用户回到之前的数据分析场景。

适合展示：

- 最近问题
- 会话时间
- 问答轮数
- 会话类型
- 继续查看按钮

### 3. 会话详情页

展示某个会话中的完整问答过程。

适合展示：

- 每轮用户问题
- Hermes 回答
- SQL 执行记录
- 查询状态
- 收藏入口

### 4. 我的收藏页

展示用户收藏过的高价值 SQL 问答。

适合展示：

- 问题摘要
- 回答摘要
- SQL 数量
- 收藏时间
- 继续追问入口

### 5. 收藏续聊页

用户基于某条收藏继续提问。系统会把收藏中的 SQL 和摘要作为上下文，让后续分析更连贯。

适合场景：

- 在已收藏 SQL 基础上换维度分析
- 在历史查询基础上加筛选条件
- 对同一类指标做二次拆解

## 六、权限与安全设计

系统通过 API Key 保护核心接口，确保只有授权前端或服务可以调用。

同时，系统围绕 `userId` 做用户数据隔离：

- 用户只能收藏自己的问答结果。
- 用户只能查看自己的收藏列表。
- 用户只能查看自己授权范围内的会话记录。
- 收藏详情支持用户归属校验。

这种设计既保证了前端调用的便利性，也避免不同用户之间的数据串扰。

对于下载类能力，下载链接使用不可猜测的唯一 ID。用户拿到链接即可下载，系统不会暴露可遍历的文件路径。

## 七、数据沉淀设计

系统会将关键数据写入 MySQL，形成长期可用的数据资产。

沉淀的数据包括：

- 用户会话
- 每轮问答
- Hermes 响应 ID
- SQL 执行记录
- 工具调用记录
- SQL 收藏记录
- 收藏关联的 SQL 明细

这些数据让系统从“即时回答工具”升级为“可持续运营的数据分析平台”。

后续可继续扩展：

- 用户常用问题分析
- 热门 SQL 模板沉淀
- 收藏推荐
- 指标口径治理
- 数据分析知识库
- 问答质量评估

## 八、业务收益

### 对业务用户

- 不懂 SQL 也能完成数据查询。
- 历史问答可以随时回看。
- 常用分析可以收藏复用。
- 可以基于已有结果继续追问。
- 减少对数据开发和分析师的重复依赖。

### 对产品团队

- 从单次问答升级为完整数据分析工作流。
- 收藏和历史能力提升用户留存。
- 可以围绕收藏、历史、续聊设计更多产品能力。
- 用户行为数据可反哺产品迭代。

### 对运营与管理团队

- 用户查询行为可追踪。
- 高价值分析结果可沉淀。
- 问题排查有完整链路。
- 后续可以识别高频指标需求和数据痛点。

### 对技术与数据团队

- SQL 执行、工具调用、问答结果都有记录。
- 异常问题更容易定位。
- 可以基于真实使用数据优化能力。
- 为权限治理、审计和质量评估提供基础。

## 九、典型使用场景

### 场景一：业务人员临时查数

用户输入“帮我查一下今天订单数”。Hermes 自动执行查询并返回结果。用户不需要知道表结构，也不需要写 SQL。

### 场景二：运营复用常用分析

运营同事经常查看某类活动数据。第一次查询满意后点击收藏，之后可以直接从收藏列表进入，不必重复描述需求。

### 场景三：基于历史结果继续分析

用户先查询“上周订单量”，再追问“按城市拆分”，系统能在同一会话里保持上下文，让分析连续推进。

### 场景四：管理者追溯分析过程

当某个数据结论需要复核时，可以打开会话详情，查看用户问题、Hermes 回答和实际执行 SQL，保证结论可追溯。

## 十、总结

这组接口共同构成 Hermes 的智能数据分析闭环。

它解决的不只是“问一个问题、返回一个答案”，而是让用户的数据分析过程可以被保存、复用和延展。

最终效果是：

- 用户提问更简单
- 数据查询更高效
- 分析过程可追溯
- 高价值 SQL 可沉淀
- 历史经验可复用
- 后续产品能力可扩展

Hermes 因此不只是一个智能问答接口，而是一个面向业务场景的数据分析基础能力。
